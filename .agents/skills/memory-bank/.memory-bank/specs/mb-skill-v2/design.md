# Memory Bank Skill v2 — SDD-ready lightweight GSD alternative

## 1. Motivation

Текущий скил покрывает планирование (`/mb plan`), верификацию (`/mb verify`), session lifecycle (`/mb start`/`/mb done`) и rules enforcement. Пробелы:

- нет этапа **discovery** требований (что есть в GSD через `discuss-phase`)
- нет **исполнительного движка** — стейджи выполняются вручную, без review-loop и без защитных gate'ов
- нет **SDD-дисциплины** — требования в планах свободной формы, не линкуются к тестам, traceability отсутствует
- нет **явного roadmap** — `plan.md` перегружен ролями (priorities + active plan + direction)
- нет **конфигурируемого pipeline'а** — правила вшиты в промпты команд

**Цель v2** — закрыть эти пробелы **без** раздувания скила до GSD-масштаба. Таргет: скил остаётся self-contained (работает без superpowers/gsd), но даёт execution engine + SDD с token footprint 15-25% от GSD на эквивалентную задачу.

## 2. Overview — четыре новые команды + 4 hook'а + pipeline.yaml

## 3. Data model — plan / spec / roadmap / backlog / traceability

Базовый принцип: **каждый артефакт отвечает на свой вопрос, пересечений нет**.

| Артефакт | Роль | Отвечает на | Lifecycle |
|----------|------|-------------|-----------|
| `specs/<topic>/` | справочник фичи | "что это за фича и как устроена" | столько же, сколько сама фича |
| `plans/<date>_<type>_<topic>.md` | плейбук итерации | "что именно делать сейчас" | до закрытия Sprint/Phase |
| `roadmap.md` *(rename from `plan.md`)* | оркестрация планов | "в каком порядке и статусе планы" | постоянный |
| `backlog.md` *(rename from `BACKLOG.md`)* | парковка | "идеи / ADR / future tasks / rejected" | постоянный, append-mostly |
| `checklist.md` | активные задачи | "что прямо сейчас в работе" | ротируется |
| `status.md` *(rename from `STATUS.md`)* | dashboard | "где мы, куда смотреть" | постоянный |
| `traceability.md` | REQ→Plan→Test матрица | "чем покрыт каждый REQ" | автогенерируемый |

Правила непересечения:
- текст REQ живёт **только** в `specs/`; plans ссылаются по ID
- задача в `checklist.md` существует **только** пока stage активен; после `/mb done` → `progress.md`
- `backlog.md` **не** содержит задачи активных спринтов — только future/ideas/ADR/rejected
- `traceability.md` — производная, регенерируется из `specs/*/requirements.md` + `plans/*.md:covers_requirements`

## 5. `/mb discuss` — requirements elicitation

**Invocation:** `/mb discuss <topic>`

**Флоу:**
1. Если `context/<topic>.md` есть → спросить "продолжить / перезаписать / отменить"
2. Прогреть контекст: `roadmap.md`, `research.md`, `codebase/*.md`
3. 5 фаз интервью, по одному вопросу за раз:
   - Phase 1 — Purpose & Users
   - Phase 2 — Functional Requirements (**EARS enforced**)
   - Phase 3 — Non-functional (performance, security, scale)
   - Phase 4 — Constraints + out-of-scope
   - Phase 5 — Edge cases & failure modes
4. После каждой фазы — проговорить найденное, валидировать
5. На финале: EARS-регекс валидация + semantic-check атомарности
6. Запись `context/<topic>.md`, regen `traceability.md`, update `status.md`

**EARS-формат (Easy Approach to Requirements Syntax) — 5 паттернов:**
- Ubiquitous: `The <system> shall <response>`
- Event-driven: `When <trigger>, the <system> shall <response>`
- State-driven: `While <state>, the <system> shall <response>`
- Optional: `Where <feature>, the <system> shall <response>`
- Unwanted: `If <trigger>, then the <system> shall <response>`

ID: **REQ-NNN сквозная** в рамках проекта (не topic-local). Нумерация через `scripts/mb-req-next-id.sh` — читает все `specs/*/requirements.md`, возвращает `max + 1`.

**Валидатор `scripts/mb-ears-validate.sh`:**
- Regex: `^\s*-\s+\*\*REQ-\d+\*\*.*(The|When|While|Where|If)\b.+\bshall\b.+`
- Semantic: LLM-check атомарности (1 поведение) + testability (можно написать assert)

**Шаблон `context/<topic>.md`:**

```markdown
---
topic: <topic>
created: YYYY-MM-DD
status: draft | ready
---

# Context: <topic>

## Purpose & Users
...

## Functional Requirements (EARS)
- **REQ-001** (ubiquitous): The system shall ...
- **REQ-002** (event-driven): When ..., the system shall ...

## Non-Functional Requirements
- **NFR-001**: ...

## Constraints
...

## Edge Cases
...

## Out of Scope
...
```

Traceability для REQ хранится не здесь (см. §10).

## 6. `/mb plan` — SDD-lite (эволюция существующей)

**Изменения в существующей команде:**
- читает `context/<topic>.md` если имя совпало или передан `--context <path>`
- с флагом `--sdd`: **блок** если `context.md` не в EARS или отсутствует
- без `--sdd`: **рекомендация** (warning в выводе), но не блок
- добавляет в шаблон опциональные секции сверх существующих:
  - `## Requirements` — список REQ-ID (не копия текста)
  - `## Design` — architecture decisions, interfaces
  - `## Tasks (Stages)` — существующие `<!-- mb-stage:N -->` блоки
- каждый Stage получает поле `covers_requirements: [REQ-001, REQ-003]`
- `mb-plan-sync.sh` после скаффолда → также триггерит `mb-roadmap-sync.sh` + `mb-traceability-gen.sh`

**Backward-compat:** старые планы без SDD-секций — оставляем как есть, не апгрейдим.

**Frontmatter плана (расширяется):**

```yaml
---
type: feature
topic: <topic>
status: in_progress | queued | paused | done | cancelled
depends_on: []
parallel_safe: false
linked_specs: [specs/<topic>]
sprint: 1
phase_of: <phase-name>
created: YYYY-MM-DD
---
```

## 7. `/mb sdd` — полноценный Kiro-style

**Invocation:** `/mb sdd <topic>`

**Создаёт:**

```
specs/<topic>/
├── requirements.md    # только EARS, REQ-NNN
├── design.md          # architecture + interfaces (Protocol/ABC)
└── tasks.md           # stages с чекбоксами
```

**Отличия от `/mb plan` (SDD-lite):**
- три файла вместо одного — позволяет параллельную работу над requirements/design/tasks
- `requirements.md` самодостаточен, совместим с Kiro/Kilo export
- `tasks.md` совместим с их чекбокс-форматом
- **жёсткая** SDD-валидация: `design.md` обязан содержать interface definitions (Protocol/ABC/interface)
- verifier требует что каждое REQ имеет implementing task **и** contract test

**Связка с `/mb discuss`:** `/mb sdd` после `/mb discuss` — `requirements.md` генерируется из `context.md` (EARS-часть копируется as-is), `design.md` и `tasks.md` — через продолжение интервью.

**Связка с `/mb plan`:** spec **дополняет** plan, не заменяет. Когда существует `specs/<topic>/` и нужен исполняемый plan — `/mb plan feature <topic>` создаёт план с секцией `## Requirements` содержащей `See: specs/<topic>/requirements.md#REQ-001..REQ-042` (ссылкой, не копией). Один spec может обслуживать несколько plans (Phase из нескольких Sprint-plans на один spec).

**`/mb work` на spec'е:** читает `tasks.md`, стейджи = чекбоксы; отмечает done по мере выполнения; `requirements.md` / `design.md` не трогает.

## 8. `/mb work` — исполнительный движок

### 8.1 Сигнатура

```
/mb work [<target>] [--range A-B] [--auto] [--max-cycles N]
         [--dry-run] [--budget TOK] [--slim | --full]
         [--allow-protected]
```

### 8.2 Резолв `<target>`

Очерёдность:
1. Существующий путь к plan-файлу → используется
2. Имя плана без пути → substring-поиск в `plans/`
3. Имя topic'а → ищется `specs/<topic>/tasks.md`
4. Свободный текст (≥3 слов) → агент читает `roadmap.md` + active plans, мэтчит по topic, подтверждает у пользователя
5. Пустое → активный план из `<!-- mb-active-plan -->` в `roadmap.md`

### 8.3 `--range A-B` — авто-детект уровня

- plan-файл → stages (`--range 2-4`)
- Phase (несколько Sprint-планов) → sprints (`--range 1-3`)
- `--range 3` = один элемент
- `--range 2-` = со 2-го до конца

### 8.4 `--auto` — автопилот

Без `--auto`: после каждого stage — резюме + confirmation prompt.

С `--auto`: летит без остановок, **кроме** hard stops:
- review-loop упёрся в `max_cycles` без verdict APPROVED
- verifier уронил stage-complete check
- попытка Write/Edit в protected path без `--allow-protected`
- `--budget` исчерпан
- `sprint_context_guard.hard_stop_tokens` достигнут (190k по умолчанию)

### 8.5 Review loop (ядро)

Per stage:

1. **Implement step** — роль auto-selected по context'у stage'а (backend/frontend/...) через `pipeline.yaml:roles`. Вызов через `Task` tool с выбранным агентом.
2. **Review step** — один `reviewer` агент получает diff + rubric из `pipeline.yaml:review_rubric`.
   - Verdict формат: `APPROVED` или `CHANGES_REQUESTED`
   - Обязательный список issue'ов с `severity: blocker | major | minor` + конкретными file:line + предложенным fix
   - Severity gate из `pipeline.yaml:severity_gate` (дефолт `blocker=0, major=0, minor≤3`)
3. **Fix step** (только если `CHANGES_REQUESTED` и cycle < max_cycles) — implement агент получает issue list, правит, возвращается в review
4. **Verify step** (после final APPROVED) — `plan-verifier` запускает 7 checks (§11)
5. **Stage-done** — если verifier PASS → mark stage done, update checklist, move to next

Counter для цикла — локальный per-stage, не persist между sessions. Если сессия прервалась на mid-cycle, восстанавливается из git + stage checkpoint marker.

### 8.6 `--slim` vs `--full` (context strategy)

- `--full` (дефолт): агенты получают полный контекст stage'а + linked spec + relevant codebase sections
- `--slim`: активируется hook `context-slim-pre-agent.sh` — агенты получают только active stage + его DoD + covered REQs + git diff staged (20-50k экономия за invoke)

`--slim` выставляет env `MB_WORK_MODE=slim` для hook'а.

### 8.7 Примеры

```bash
/mb work                                # активный план, интерактив, все stages
/mb work --auto                         # активный план, автопилот
/mb work auth-refactor --range 2-4      # plan "auth-refactor", stages 2-4
/mb work billing --auto --range 1-3     # phase billing, sprints 1-3, автопилот
/mb work "починить флаки в auth"        # фристайл → резолвит через мэтчинг
/mb work --dry-run                      # показать план выполнения
/mb work --budget 150000 --auto         # стоп при 150k spent
```

## 9. `pipeline.yaml` — схема

```yaml
version: 1

# ===== Роли — маппинг на агентов, плагин-стиль per-role overrides =====
roles:
  developer:  { agent: mb-developer }
  backend:    { agent: mb-backend,  fallback: mb-developer }
  frontend:   { agent: mb-frontend, fallback: mb-developer }
  ios:        { agent: mb-ios,      fallback: mb-developer }
  android:    { agent: mb-android,  fallback: mb-developer }
  architect:  { agent: mb-architect }
  devops:     { agent: mb-devops,   fallback: mb-developer }
  qa:         { agent: mb-qa,       fallback: mb-developer }
  analyst:    { agent: mb-analyst }
  reviewer:
    agent: mb-reviewer
    override_if_skill_present:
      skill: superpowers
      agent: superpowers:requesting-code-review
  verifier:   { agent: plan-verifier }

# ===== Pipeline per stage =====
stage_pipeline:
  - step: implement
    role: auto               # auto-select по context'у stage'а
    tdd: true
  - step: review
    role: reviewer
    categories: [logic, code_rules, security, scalability, tests]
    severity_gate:
      blocker: 0
      major: 0
      minor: 3
    max_cycles: 3
    on_max_cycles: stop_for_human  # | continue_with_warning
  - step: verify
    role: verifier
    checks:
      - ears_format
      - contract_first_tests
      - test_red_green
      - dod_checkboxes
      - architecture_drift
      - no_placeholders
      - protocol_abc_defined

# ===== Budget / risk =====
budget:
  default_limit: null          # null = unlimited; CLI --budget overrides
  warn_at_percent: 80
  stop_at_percent: 100

protected_paths:
  - ".env*"
  - "ci/**"
  - ".github/workflows/**"
  - "Dockerfile*"
  - "k8s/**"
  - "terraform/**"

sprint_context_guard:
  soft_warn_tokens:  150000
  hard_stop_tokens:  190000

# ===== Review rubric — fixed keys, extensible items =====
review_rubric:
  logic:
    - "Every EARS requirement has at least one assertion in tests"
    - "Edge cases from requirements covered"
  code_rules:
    - "SOLID: SRP — files <300 lines OR <=3 public methods"
    - "No placeholders (TODO/...)"
    - "Imports complete, functions copy-paste ready"
  security:
    - "Input validation at boundaries"
    - "No secrets in code"
    - "No raw SQL string concat"
  scalability:
    - "No N+1 queries introduced"
    - "Async where IO-bound"
  tests:
    - "Contract-first: Protocol/ABC defined before impl"
    - "Integration tests > unit tests (Testing Trophy)"
    - "No test.skip / describe.skip"

# ===== SDD enforcement =====
sdd:
  require_ears_in_sdd_command: true     # /mb sdd — hard requirement
  require_ears_in_plan_command: false   # /mb plan — soft (warning)
  require_ears_in_plan_with_sdd_flag: true  # /mb plan --sdd — hard
  covers_requirements_policy: warn      # warn | block | off
  full_mode_path: ".memory-bank/specs"
```

**Fallback:** если проектного `pipeline.yaml` нет — грузится `references/pipeline.default.yaml` (этот самый). `/mb config` копирует дефолт в проект при первом редактировании.

**Валидация** — `scripts/mb-pipeline-validate.sh` (JSON-schema check). Вызывается из `/mb work` на старте и из `/mb doctor`.

### 10.3 `backlog.md` (rename from `BACKLOG.md`)

Разделы сохраняются:
- `## Ideas (I-NNN)` — идеи без плана
- `## ADR (ADR-NNN)` — принятые решения
- `## Future tasks` — то что запланировано в roadmap но plan ещё не создан
- `## Rejected` — с причиной

**Не дублирует** активные-спринт-задачи (те в `checklist.md`). Не дублирует ADR из `specs/*/design.md` (те — частные проектные решения фичи; ADR — cross-cutting).

## 11. Verifier — расширение `plan-verifier.md`

Существующий агент, промпт расширяется. **7 checks**, все запускаются на stage-complete:

1. **EARS format compliance** — regex + semantic для каждого REQ из `covers_requirements`. Отсутствие REQ-ссылки при наличии `## Requirements` → severity = **warn** / **block** / **off** по `covers_requirements_policy`.

2. **Contract-first tests existence** — для каждого REQ должен быть test ассертящий поведение. Поиск: grep в `tests/` по REQ-NNN в docstring/test-name. Отсутствие → blocker.

3. **Test-first discipline (TDD)** — `git log --follow` на тест-файл: commit теста ≤ commit импла (или в одном коммите). Если test commit > impl commit на >1 коммит → WARNING (legit refactor кейсы).

4. **Protocol/ABC existence** — в `design.md` или `## Design` plan'а — explicit interface. Для impl — `class X(Protocol)` или equivalent. Опциональный (не всегда применим к UI).

5. **DoD completeness** — все `- [ ]` → `- [x]`. SMART-валидация: measurable verb.

6. **Architecture drift** — через `codebase/graph.json`: edges `domain/** → infrastructure/**` → blocker.

7. **No placeholders** — grep по изменённым файлам: `TODO`, `FIXME`, `\.\.\.` в function body, `pass  # stub`, `throw new Error("not implemented")`. Исключение: рядом комментарий `staged stub behind feature flag <name>`.

**Output:** `reports/YYYY-MM-DD_verify_<topic>.md`

```markdown
# Verify: <topic> Sprint N

## Summary
- ✅ EARS compliance: 42/42
- ✅ Contract-first tests: 40/42 (2 WARN)
```

## 13. Hooks — 4 критичных

| # | Hook | Type | Назначение |
|---|------|------|------------|
| 1 | `protected-paths-guard.sh` | PreToolUse (Write/Edit) | Блок записи в `.env` / CI / infra без `--allow-protected` (env `MB_ALLOW_PROTECTED=1`) |
| 2 | `plan-sync-post-write.sh` | PostToolUse (Write in `plans/**` or `specs/**`) | Триггерит chain: `mb-plan-sync.sh` → `mb-roadmap-sync.sh` → `mb-traceability-gen.sh` → `mb-status-refresh.sh` |
| 3 | `ears-pre-write.sh` | PreToolUse (Write in `specs/*/requirements.md` or `context/*.md`) | Прогон через `mb-ears-validate.sh`; невалидный EARS не сохраняется |
| 4 | `context-slim-pre-agent.sh` | PreAgentInvoke (role-agent из `pipeline.yaml:roles`) | При `MB_WORK_MODE=slim` — подменяет prompt на минимальный контекст (active stage + DoD + covered REQs + git diff). Noop при `MB_WORK_MODE=full` (дефолт) |

**Не-hook скрипты** (вызываются явно из команд):
- `mb-checklist-auto-update.sh` — из `/mb done`
- `mb-status-refresh.sh` — chained из hook #2
- `mb-budget-meter.sh` — из `/mb work` loop

## 14. `/mb config` — мастер `pipeline.yaml`

**Invocation:** `/mb config [--show | --reset | --edit <key>]`

- без аргументов → интерактивный мастер (копирует дефолт → идёт по разделам)
- `--show` → merged effective config (default + project overrides)
- `--edit severity_gate` → точечное редактирование
- `--reset` → сброс к дефолту, бэкап в `pipeline.yaml.bak-<timestamp>`

**Блоки мастера:**
1. Установлен ли superpowers → override reviewer?
2. Severity gate thresholds per category
3. max_cycles, on_max_cycles
4. Protected paths (add/remove)
5. Budget default
6. SDD enforcement (require_ears_*)
7. Review rubric — custom-пункты (add/remove)
8. Per-role agent overrides

Валидация после save — `mb-pipeline-validate.sh`. Провал → откат к бэкапу + error.

## 15. Migration from v1 → v2

**Скрипт `scripts/mb-migrate-v2.sh`:**

- идемпотентный, с `--dry-run`
- создаёт бэкап `.memory-bank/.migration-backup-<timestamp>/`
- операции:
  1. rename: `STATUS.md` → `status.md`, `BACKLOG.md` → `backlog.md`, `RESEARCH.md` → `research.md`, `plan.md` → `roadmap.md`
  2. `roadmap.md` content transform:
     - existing `<!-- mb-active-plan -->` → секция `## Now (in progress)`
     - existing priorities text → `## See also` как note
     - add empty sections: `## Next`, `## Parallel-safe`, `## Paused / Archived`, `## Linked Specs`
  3. grep+sed по всем `.md` в `.memory-bank/`: update ссылок на старые имена
  4. `mb-roadmap-sync.sh` чтобы заполнить roadmap актуальными данными

**Autodetect:** `/mb start` и `/mb context` детектят наличие `STATUS.md` → предлагают миграцию с diff-preview. Если отказ → скрипты читают старые имена (backward-compat).

**Период совместимости:** 2 версии скила поддерживают оба именования. После — `/mb doctor` даёт WARN, ещё через версию — ERROR без миграции.

**Рефакторинг кода скила (разовый PR):** все ссылки на старые имена в `commands/`, `references/`, `scripts/`, `agents/`, `SKILL.md`, `README.md`, `CLAUDE.md`.

## 17. Token efficiency vs GSD — целевые цифры

| Механизм | GSD | Skill v2 | Экономия за итерацию |
|----------|-----|----------|----------------------|

| Review loop | 4 параллельных reviewer-агента | 1 reviewer с rubric + severity gate | ~30-60k / цикл |
| Между стейджами | перечитывает full plan | изолирован, только свой блок + traceability | ~15k / stage |

## 20. Implementation phasing (4 фазы)

Каждая фаза = отдельный plan-файл в `plans/`, укладывается в 200k токенов одного Sprint'а (или Phase из Sprint'ов при надобности).

**Phase 1 — Foundation** (rename + roadmap + traceability)
- Sprint 1: `mb-migrate-v2.sh` + обновление существующих команд на новые имена
- Sprint 2: `roadmap.md` format + `mb-roadmap-sync.sh` + `mb-traceability-gen.sh`
- **Gate:** `/mb start` работает на новых именах, `roadmap.md` живой, существующие тесты зелёные

**Phase 2 — Discussion & SDD artifacts**
- Sprint 1: `/mb discuss` + EARS validator + `context/<topic>.md` template
- Sprint 2: `/mb sdd` + `specs/<topic>/` структура + адаптация `/mb plan` (SDD-lite секции)
- **Gate:** проход discuss → plan → verify с EARS, traceability.md собирается

**Phase 3 — Work engine**
- Sprint 1: `pipeline.yaml` schema + `/mb config` мастер + `pipeline.default.yaml`
- Sprint 2: `/mb work` базовый (без review-loop) + ролевые агенты (9 шт.)
- Sprint 3: review-loop + severity gates + verdict persistence + reporter
- **Gate:** `/mb work <plan>` проходит полный stage: implement → review → verify → done

**Phase 4 — Hardening**
- Sprint 1: `plan-verifier` расширение (7 checks) + hooks (4 критичных)
- Sprint 2: `--auto` / `--range` / `--budget` / `--allow-protected` + sprint_context_guard
- Sprint 3: superpowers overrides + документация + installer update
- **Gate:** `/mb work --auto --range 1-3` отрабатывает phase целиком без участия пользователя

## 21. Open questions / risks

- **Semantic EARS check** — LLM-вызов при каждом Write в requirements.md может быть медленным. Мitigation: hook вызывает только regex; semantic — в явной команде `/mb verify` или `/mb doctor`.
- **`context-slim-pre-agent.sh`** — маскирует что агент реально получил; сложно дебажить. Mitigation: `--slim` не дефолт; явный log в `.memory-bank/.slim-context-log/` с полным snapshot'ом подставленного prompt'а.
- **Rename-migration regressions** — старые ссылки в user-скриптах/external docs могут сломаться. Mitigation: 2-версии compat window + WARN в doctor.
- **Subagent copy-upstream drift** — оригинальные агенты в `~/.claude/agents/` эволюционируют; наши копии устаревают. Mitigation: скрипт `scripts/mb-agents-sync-upstream.sh` — сравнивает копии с upstream, показывает diff, опционально применяет (при подтверждении).
